# Triage Summary

**Title:** Description search regression in v2.3.0: full-text search likely falling back to full table scan on SQLite

## Problem
Since upgrading from v2.2 to v2.3.0, searching task descriptions takes 10-15 seconds per query. Title search remains fast. The issue occurs on a self-hosted instance with ~5,000 tasks containing large description fields (full meeting minutes, thousands of words each). CPU usage spikes visibly during searches.

## Root Cause Hypothesis
The v2.3.0 release likely changed the description search implementation — either removing SQLite FTS (full-text search) indexing, failing to migrate the FTS index during upgrade, or switching to a naive LIKE/%pattern% scan against the raw description column. This would cause SQLite to perform a sequential scan of all 5,000 task descriptions on every query, explaining the uniform ~10-15s latency regardless of query complexity and the CPU spike.

## Reproduction Steps
  1. Set up a self-hosted TaskFlow v2.3.0 instance with default (SQLite) configuration
  2. Populate with a large number of tasks (~5,000) where many have long description fields (1,000+ words)
  3. Perform a single-word search using the main search bar
  4. Observe that title-matching results return quickly but description-matching results take 10-15 seconds
  5. Monitor CPU usage during description search — expect a spike

## Environment
TaskFlow v2.3.0, self-hosted on laptop, default installation (SQLite database), ~5,000 tasks with large description fields

## Severity: high

## Impact
Affects any self-hosted user with a non-trivial number of tasks who searches descriptions. Power users with large datasets (like this reporter) are most affected. Search is a core workflow — 10-15s latency per query significantly degrades usability. No known workaround other than avoiding description search.

## Recommended Fix
1. Diff the search query logic between v2.2 and v2.3.0 — look for changes to how description search queries are constructed (FTS vs LIKE). 2. Check whether the v2.3.0 migration preserves or rebuilds the SQLite FTS virtual table for descriptions. 3. If FTS was removed, restore it. If the migration fails to rebuild the index, add a migration step. 4. Consider adding EXPLAIN QUERY PLAN logging for search queries in debug mode to catch future regressions.

## Proposed Test Case
Create a test with 5,000+ tasks with description fields of 1,000+ words each on SQLite. Assert that a description search query completes in under 2 seconds. Run this test against both v2.2 (baseline) and v2.3.0 (regression) to confirm the performance delta.

## Information Gaps
- Exact search query SQL generated by v2.3.0 (requires code inspection, not reporter input)
- Whether the SQLite FTS index exists in the reporter's database file (could be checked but wouldn't change the fix direction)
- Whether other self-hosted users on v2.3.0 with smaller datasets also experience slower description search (just less noticeably)
